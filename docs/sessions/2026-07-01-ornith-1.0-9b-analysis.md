# 2026-07-01 — Ornith 1.0 9B results.tsv analysis + improvement ideas

## Goal

Analyze the existing `results.tsv` data for `ornith-1.0-9b-Q4_K_M.gguf`, identify why its `val_score` lags the best 9B rival, and document concrete, low-risk ideas to close the gap without running new benchmarks in this session.

- `llama-bench` tg throughput: ~45–54 t/s

## Findings

1. **No AutoLoop data exists for Ornith 9B.** All 22 rows are manual single runs / validation runs. The model has never been through the systematic hill-climbing loop that `ornith-1.0-35b` already went through.
2. **Best comparable 10-task score is 0.580**, vs. **Qwythos-9B 0.640** — a ~0.06 gap on the same task budget.
3. **BigCode is the weakest dataset** for Ornith 9B: best observed BigCode pass@1 is only 0.2 in the 10-task keep row, and often 0.0.
4. **LCB tops out at 0.5** in validation but is only 0.4 in the best 10-task row; LCB variance is high.
5. **HE and MBPP are strong** (0.8–0.9 and 0.9 respectively in the best 10-task run), suggesting the model understands basic coding but struggles with harder / library-heavy tasks.
6. **All successful runs use `kv=q4_0` and `ctx=131072`.** KV precision and context size have not been varied for this model in the log.
7. **Validation runs score higher** (up to 0.625) but use only 2 tasks per dataset, so they are noisier and not directly comparable to 10-task rows.

## Improvement ideas ( ranked by expected impact )

1. **Run `autoloop.py` on `ornith-1.0-9b-Q4_K_M.gguf`.** This is the single biggest lever; 35B already benefitted from automated single-parameter sweeps.
2. **Vary KV cache precision independently.** Current baseline is K=V=q4_0. Candidates:
   - K=q8_0 / V=q4_0
   - K=q8_0 / V=q8_0
   - K=f16 / V=q4_0 (only if VRAM stays ≤ 7.9 GB)
3. **Optimize generation parameters for pass@1 coding.** Current config is `TEMP=0.4`, `TOP_P=0.95`. Try:
   - `TEMP=0.0–0.2`
   - `TOP_P=0.9`
   - `TOP_K=20–40`
   - `MIN_P=0.0`
   - `REPEAT_PENALTY=1.0` or `None`
   - `PRESENCE_PENALTY=None` / `0.0`
4. **Toggle `CONT_BATCHING`.** The search space includes `True`/`False`; no evidence either way for Ornith 9B yet.
5. **Try `JINJA=True`.** If the model is instruction-tuned, correct chat templating can improve instruction following and code extraction.
6. **Batch / micro-batch grid.** Test:
   - `BATCH_SIZE=512`, `UBATCH_SIZE=128`
   - `BATCH_SIZE=1024`, `UBATCH_SIZE=256`
   These affect both TPS and, in some cases, output determinism.
7. **BigCode-specific focus.** Since BigCode is the lowest component:
   - confirm `BIGCODE_MAX_TOKENS=2048` is used
   - eliminate repeat/presence penalties
   - verify the model emits a single fenced code block and that `task_func` is defined
8. **Increase task limit for final confirmation.** Once a promising config is found, re-run with `CODING_TASK_LIMIT=30` or full to reduce the variance seen between 10-task and validation rows.
9. **If available, test a Q5_K_M or MTP variant.** Quantization/precision and MTP can move coding scores more than runtime flags for a dense 9B model.
10. **Keep `CTX_SIZE` at 131072 and `FLASH_ATTN=on`.** These are already optimal per repo contract and prior evidence.
