1. Mini-glossário

Motor
Software que carrega os pesos e gera tokens (ex.: llama.cpp).
Modelo
Os pesos no disco — tipicamente um arquivo .gguf.
Quant
Compressão dos pesos (Q4, Q5, Q8…). Menos bits → menos VRAM, costuma aumentar o TPS.
VRAM
Memória da GPU onde o motor tenta colocar pesos + KV. O que não couber pode ir para RAM/CPU (offload) — especialmente útil em MoE (Dia 3).
KV / contexto
Memória do histórico da conversa. Mais contexto = mais KV = mais memória.
TPS
Tokens por segundo — métrica de velocidade nesta semana de performance.
API / harness
API local (estilo OpenAI) em 127.0.0.1; harness = script, app ou agente que se comunica com essa API.

Arquivos GGUF ficam na sua máquina — sem commits no git. Glossário completo: reference/glossario.html.

2. Motores de inferência

Um motor carrega o modelo e gera os tokens. Um modelo (ex.: arquivo GGUF) são os pesos. Um quant (Q4, Q5, Q8…) é a compressão desses pesos — afeta a VRAM e o TPS.

Existem vários motores de inferência. O conceito fundamental é: carregou o modelo, ativou o servidor local, a API local fica disponível. No Dia 2 praticamos esse modelo mental diretamente no terminal com llama.cpp neste repositório.

3. Onde baixar e como escolher (foco em performance)

  • Onde: Hugging Face Hub — repositórios *-GGUF / arquivos .gguf. No fluxo deste repo, preferimos a CLI hf.
  • Como escolher: cabe na VRAM? Quant mais baixo → menos VRAM, tipicamente mais TPS. Ignore modelos gigantes se você tem 8 GB de VRAM.
  • Não escolha por marketing de benchmark de qualidade nesta semana.

4. Usar o modelo via API (qualquer aplicativo)

Ative o Servidor Local (API compatível com o padrão OpenAI). Qualquer aplicativo ou script (harness) aponta para a URL local e o nome do modelo servido.

O conceito principal do Dia 1: com o servidor local ativo, qualquer aplicativo na sua máquina pode se comunicar com a IA local.
