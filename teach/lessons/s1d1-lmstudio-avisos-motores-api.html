1. Mini-glossário

Motor
Software que carrega pesos e gera tokens (ex.: LM Studio, llama.cpp).
Modelo
Os pesos no disco — tipicamente um ficheiro .gguf.
Quant
Compressão dos pesos (Q4, Q5, Q8…). Menos bits → menos VRAM, costuma subir TPS.
VRAM
Memória da GPU onde o motor tenta pôr pesos + KV. O que não cabe pode ir para RAM/CPU (offload) — especialmente útil em MoE (Dia 3).
KV / contexto
Memória do histórico da conversa. Mais contexto = mais KV = mais memória.
TPS
Tokens por segundo — métrica de velocidade nesta semana de performance.
API / harness
API local (estilo OpenAI) em 127.0.0.1; harness = script, app ou agente que fala com essa API.

GGUF fica na máquina — não commits no git.

2. Motores de inferência

Um motor carrega o modelo e gera tokens. Um modelo (ex. ficheiro GGUF) são os pesos. Um quant (Q4, Q5, Q8…) é a compressão desses pesos — afeta VRAM e TPS.

Existem vários motores (LM Studio, llama.cpp, outros…). Dia 1 usamos LM Studio porque a UI deixa óbvio: carregaste o modelo, ligaste o servidor, a API está viva. Dia 2 repetimos o mesmo mental model com llama.cpp neste repositório.

3. Onde baixar e como escolher (lente performance)

  • Onde: Hugging Face Hub — repos *-GGUF / ficheiros .gguf. No Dia 2, no fluxo deste repo, preferimos hf CLI.
  • Como escolher (hoje): cabe na VRAM? Quant mais baixo → menos VRAM, tipicamente mais TPS. Ignora o ego do “70B” se tens 8 GB.
  • Não escolhas pelo marketing de benchmark de qualidade nesta semana.

4. Usar o modelo via API (qualquer harness)

No LM Studio: carrega o modelo → ativa o Local Server (API compatível com estilo OpenAI). Qualquer harness (script, app, agente) aponta para a base URL local e o nome do modelo servido.

O truque pedagógico do Dia 1: se vês o server “on” na UI, entendeste o contrato que o Dia 2 vai repetir com llama-server — só muda a embalagem.

Docs: lmstudio.ai/docs (secção Local Server / API).
